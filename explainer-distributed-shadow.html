The problem: one simulator, too much world

Shadow is a deterministic network simulator. That means it runs real applications, but inside a controlled simulated network where time, packets, sockets, and host behavior are repeatable. Distributed Shadow asks a simple question: what if one physical machine is not enough?

01

Vanilla Shadow

One Shadow process owns the whole simulated world. It can use many worker threads, but all simulated hosts live inside one operating-system process on one machine.

02

Distributed Shadow

Several Shadow processes each own a slice of the simulated hosts. We call each slice a shard: a partition of the simulated world.

03

The hard part

If host A on shard 0 sends a packet to host B on shard 3, that packet must cross a real machine boundary while still arriving at the correct simulated time.

Distributed Shadow is not about making the simulator less careful. It is about making the careful simulator fit across more hardware.

A useful analogy: imagine a giant model train city. Vanilla Shadow is one conductor controlling every train from one desk. Distributed Shadow gives four conductors four districts, but they still need one clock and a rulebook so trains crossing district borders do not crash.

Why we took this approach

The starting point was not, "let us add MPI because MPI is cool." The starting point was a practical bottleneck: simulations like the 64-node ethlambda scenario are heavy, and we wanted to know if more machines could help.

chose

Shard by simulated host

Each MPI rank owns a subset of hosts. That keeps most existing Shadow internals intact: local events, local queues, local sockets, local workers. Only cross-shard packets become special.

chose

Exchange packet-arrival events

We do not ship entire process state between machines. We ship timestamped packet events. That is the smallest useful message: "deliver this packet to host B at simulated time T."

Mental model: postal districts

Each shard is a postal district. Mail within the district is cheap. Mail to another district goes through the central sorting office. MPI is that sorting office. The global simulated clock is the delivery schedule everyone must obey.

SID

ShardId

A small identifier saying which simulated district owns a host. It is the address label for distributed routing.

PM

PartitionMap

The table that maps hostnames to shards. In the 64-node run, we used explicit partitions: 16 hosts per rank.

RPE

RemotePacketEvent

A serialized promise: this packet should appear at the destination host at this simulated delivery time.

MPI

Collectives

All ranks participate together. This avoids the classic "you wait for me while I wait for you" deadlock.

The roads we did not take

The rejected designs are where the lesson gets sharp. Distributed simulation is full of ideas that sound fine until the timing model bites back.

rejected

Just run vanilla Shadow harder

Vanilla Shadow already supports parallelism. For our comparison, vanilla used one 16-core, 64GB machine with --parallelism 16. But one process still owns all memory, all hosts, and all queues. If the goal is to use multiple physical machines, thread-level parallelism is not enough.

rejected

Naive blocking send and receive

Ordered point-to-point sends look natural: shard 0 sends to shard 1, shard 1 sends to shard 2, and so on. In practice, that can deadlock. Everyone can end up waiting for someone else to receive. MPI collectives solved this by making the exchange a coordinated group operation.

rejected

Invent a custom TCP protocol first

We could have opened sockets between Shadow processes and designed our own message framing, retry behavior, ordering rules, and synchronization protocol. That would be educational, but also a lot of new failure surface. MPI already gives us the exact shape we need: launch a group of processes, give each one a rank, and make them participate in coordinated communication rounds.

rejected

Route by host ID modulo rank count

Modulo routing is tempting because it is simple. But explicit partition files let us say exactly where each host lives. The fix was to route through OutboundRemotePacket.dst_shard, derived from the configured partition map, not a guess.

rejected

Assume a shared filesystem fixes everything

Shared paths help with inputs, but they do not move live packets at simulated time. We still need a real distributed protocol for packet events and global time coordination.

[protocol]

The protocol: run, exchange, agree, repeat

A distributed simulator lives by one rule: nobody may run so far ahead that a late remote packet would have needed to arrive in the past.

phase 1

Run a safe local window

Each shard runs its local hosts until the current window end. Local packets go straight into local event queues.

phase 2

Serialize remote packets

If a packet belongs to another shard, Shadow turns it into a remote packet event with source host, destination host, delivery time, and payload.

phase 3

Exchange through MPI

MPI_Alltoall exchanges batch sizes. MPI_Alltoallv exchanges the actual payload bytes. Every rank enters the same collective operation.

phase 4

Find the next global event

MPI_Allreduce(MPI_MIN) finds the earliest next event across all shards. That becomes the start of the next window.

// The optimization we added in spirit:
worker_shared.send_remote_packets(remote_packet_exchange)?;

if remote_packet_exchange.requires_send_complete_synchronization() {
    controller.remote_packet_send_complete()?;
}

worker_shared.receive_remote_packets(remote_packet_exchange)?;

The subtle bit: for MPI, the packet exchange already synchronizes all ranks. So an extra post-send barrier was not buying correctness. It was buying waiting.

MPI, without the fog machine

MPI stands for Message Passing Interface. It is a standard API for programs that run as many cooperating processes and exchange messages. Think of it as the radio protocol used by a fleet of ships: every ship has a call sign, everyone knows who is in the fleet, and there are standard maneuvers for broadcasting, gathering, and synchronizing.

RANK

Ranks are identities

An MPI program starts multiple copies of the same executable. Each copy gets a rank, usually numbered 0, 1, 2, and so on. In Distributed Shadow, rank 0 owns shard 0, rank 1 owns shard 1, and so on.

WORLD

A communicator is the room

MPI_COMM_WORLD is the default group containing all ranks in the job. When we call a collective on this communicator, every Shadow shard in the run participates.

COLL

Collectives are group moves

A collective operation is not one process talking to one process. It is the whole group doing one coordinated move: everyone checks in, exchanges data, or agrees on a result.

SYNC

Progress is coordinated

MPI does not simulate the network for us. Shadow still owns simulated time. MPI only moves real bytes between real processes so Shadow can preserve the simulated ordering.

MPI launch model

mpirun -np 4 shadow ...
          |
          +--> process A: rank 0, shard 0, owns hosts 0..15
          +--> process B: rank 1, shard 1, owns hosts 16..31
          +--> process C: rank 2, shard 2, owns hosts 32..47
          +--> process D: rank 3, shard 3, owns hosts 48..63

All ranks run the same binary. Their rank tells them which slice of the world to own.

MPI is not a database, not a queue, and not magic shared memory. It is a disciplined way for a fixed group of processes to talk in lockstep.

How the MPI operations map to Shadow

Distributed Shadow uses a small vocabulary of MPI operations. Each one has a job in the simulation rhythm. Once you know these four verbs, the backend becomes much less mysterious.

BAR

MPI_Barrier

A barrier means: nobody leaves until everybody arrives. We still use it at startup. We removed the redundant post-packet-exchange barrier because MPI_Alltoall and MPI_Alltoallv already make every rank participate.

MIN

MPI_Allreduce(MIN)

Each shard reports its earliest next local event time. MPI returns the global minimum to every shard. That tells everyone where the next safe simulation window begins.

SIZE

MPI_Alltoall

Every shard tells every other shard how many bytes it plans to send. This is the envelope count before mailing the packages. The receiver needs these sizes before it can allocate receive buffers.

DATA

MPI_Alltoallv

The v means variable-sized. Shard 0 might send 10 bytes to shard 1 and 80,000 bytes to shard 3. Alltoallv handles uneven payloads, which is exactly what packet traffic looks like.

Why MPI was the right first distributed backend

The important word is first. MPI was not chosen because it is the only possible answer. It was chosen because it let us test the real distributed-simulation question without first building an entire networking substrate from scratch.

chose

Use MPI collectives for correctness-shaped communication

Shadow naturally runs in rounds: execute a window, exchange remote events, agree on the next time. MPI collectives are also round-shaped. That fit reduced design risk.

chose

Let mpirun handle process placement

We did not need to write a cluster launcher. OpenMPI already starts ranks across hosts using a hostfile, wires them together, and gives each process its rank and world size.

cost

Collectives can become the bottleneck

MPI made the backend correct and simple enough to validate, but it also made the cost visible. In the optimized 64-node run, MPI_Alltoall size exchange became the largest remaining MPI timing bucket.

cost

Portability is not free

The current backend binds OpenMPI exported symbols directly. That worked for our cluster, but a production-quality backend should add a small C shim or generated bindings so MPICH and other MPI implementations are cleaner targets.

Why not message queues?

A queue like Kafka or RabbitMQ is great when producers and consumers are loosely coupled. Shadow shards are tightly coupled by simulated time. We need every rank to finish the same exchange round before time advances. MPI's collective model is closer to the simulator's heartbeat.

[debug]

The messy parts that taught us something

This was not a clean march from design doc to victory. The simulator told us where our assumptions were wrong.

BUG

Wrong-shard packets

We saw errors like "host belongs to shard 0, not 1." That exposed the danger of deriving routes from host ID modulo rank count instead of the real partition map.

WAIT

Deadlock-shaped exchange

The old ordered blocking exchange could stall. Switching to MPI_Alltoall plus MPI_Alltoallv made every rank participate in the same rhythm.

ARG

Zero-byte MPI buffers

When we stopped serializing empty batches, OpenMPI rejected zero-length buffer pointers in MPI_Alltoallv. The fix was tiny but important: use dummy one-byte buffers when totals are zero.

TAX

The hidden collective tax

After removing the extra barrier, the next bottleneck became obvious: MPI_Alltoall size exchange dominated aggregate MPI time in the optimized 64-node run.

The optimization: remove waiting that did not protect correctness

The core improvement was not exotic. It was the kind of performance win experts love: delete a redundant synchronization point, but only for the backend where it is provably redundant.

evolved

Backend-specific synchronization

Socket-based exchange may still need an external post-send sync. MPI does not, because its collectives already force every rank through the same exchange.

chose

Skip empty payload serialization

Empty remote-packet groups no longer become encoded empty packet batches. That reduced payload exchange and encode/decode work while preserving the size exchange needed to discover incoming packets.

The current frontier is clear: in the optimized 64-node distributed run, aggregate MPI_Alltoall size-exchange time was about 2,224 seconds across ranks. That does not mean the wall clock was 2,224 seconds; it means ranks collectively spent a lot of waiting time in that operation. The next serious improvement likely needs fewer synchronization rounds or a smarter exchange protocol — but the single biggest win came from a different direction entirely: the scheduling window itself.

[runahead]

The bigger lever: the runahead window

The MPI tuning mattered, but the real unlock was not in MPI at all. It was in how far each shard may run before it must stop and synchronize — the runahead window.

evolved

Runahead = smallest inter-host latency

Within one window every shard runs its hosts in parallel. To stay causal, the window can be at most the smallest latency between two distinct hosts — then any cross-host packet always lands in a future window. Bigger window means fewer synchronization rounds, which means faster.

chose

Exclude self-loops: use_host_pair_runahead

The only sub-12ms edges in our topology are self-loops (a host's own loopback / local API, ~1ms). Those constrain one host's internal traffic, not cross-shard scheduling. Excluding them grows the window from ~1ms to ~12ms — roughly 12× fewer rounds. The value is computed automatically from each topology; it is never hardcoded.

One flag, two regimes

use_host_pair_runahead: true (default) gives the fast ~12ms cadence. Set it false to include self-loops and use the ~1ms window — exactly matching vanilla Shadow's timing: slower, but bit-for-bit reproducible (see below). A separate guard rejects use_dynamic_runahead in distributed mode, where a per-shard shrinking window would break cross-shard ordering.

[determinism]

Can distributed reproduce vanilla, bit for bit?

The deepest question for a deterministic simulator: does sharding change the answer? After a long hunt the result is clean — distributed, single-process, and upstream vanilla Shadow produce byte-identical output, once one scenario setting is fixed.

gotcha

The real determinism breaker: native_preemption_enabled

Generated scenarios set native_preemption_enabled: true. Shadow's own docs say it "breaks simulation determinism": it interrupts long pure-CPU code (here, hash-sig signing) based on real CPU time and advances simulated time by a fixed amount — inherently non-repeatable. It is a stock Shadow option, so even vanilla single-process runs diverged run-to-run with it on. Not an application bug, and not a distributed bug; the cross-shard machinery was correct all along.

Recipe for reproducible distributed runs

Set native_preemption_enabled: false, model_unblocked_syscall_latency: false, and use_host_pair_runahead: false (to match vanilla's ~1ms window), and use the single-threaded build of your application. Then distributed == single-process == vanilla, every time. Leave use_host_pair_runahead at its default if you only need distributed-vs-single consistency plus the fast cadence — that holds regardless of the flag.

The thing that looked like a distributed bug was a real-time preemption knob in the scenario file.

What an expert notices

AHA

Determinism is the real product

The goal is not merely to move packets quickly. The goal is to move packets while preserving repeatable event ordering. That constraint shapes every design choice.

CUT

The partition is a performance feature

If most traffic crosses shard boundaries, MPI becomes hot. A good partition minimizes the "cut" between shards, like drawing city districts so commuters do not all cross the same bridge.

CLK

Time synchronization is expensive

Every global agreement is a tax. The simulator pays it in exchange for correctness. Experts ask: can we safely agree less often?

FIT

Benchmark fit matters

This 64-node workload fit a 16-core, 64GB vanilla machine well enough that distributed overhead outweighed the benefit. A larger workload may cross that line.

Things I would tell a friend before they try this

pitfall

Do not benchmark only the happy path

A tiny 4-node run proved correctness and showed barrier counts dropping, but wall-clock time was too short and noisy. The 64-node run told the real performance story.

pitfall

Keep absolute paths identical across hosts

Every rank needs to find the same config inputs, genesis files, topology file, and binary paths. A distributed simulator is allergic to "it works on my machine" path drift.

pitfall

Watch the collectives, not just CPU usage

CPU graphs can look calm while ranks are waiting inside MPI. The useful metrics were mpi_alltoall_sizes_time_ns, mpi_alltoallv_payload_time_ns, and barrier/allreduce counters.

[transfer]

Lessons you can reuse elsewhere

Distributed Shadow is a network simulator story, but the principles travel well.

DB

Databases

Sharding a database has the same shape: local queries are cheap, cross-shard joins are expensive, and partitioning strategy can matter more than raw machine count.

ML

Machine learning

Distributed training can be slower than single-machine training when gradient synchronization dominates. More GPUs do not help if the all-reduce is the bottleneck.

OPS

Production systems

Microservices have a similar trap: splitting a monolith can create network calls where function calls used to be. The architecture must earn the new communication cost.

The portable rule

Distribution helps when the work you move apart is bigger than the coordination you add. If coordination grows faster than useful work, you built a very impressive waiting machine.
